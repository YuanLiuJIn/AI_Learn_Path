## 1. 从 GPT 到 ChatGPT：三阶段能力叠加

语言模型演进可以用三句话理解：

```text
预训练：让模型“博学”
SFT：让模型“听指令”
RLHF / GRPO：让模型“符合人类偏好”
```

### 1.1 预训练：学语言和世界知识

预训练目标通常是下一个 token 预测：

```text
给定 x_1, x_2, ..., x_t，预测 x_{t+1}
```

这让模型学到：

```text
语法
常识
事实知识
代码模式
推理模式的初步影子
```

但预训练模型不会天然“听话”。它只是会续写。

### 1.2 SFT：把“续写模型”变成“指令模型”

SFT 数据形态：

```json
{
  "instruction": "判断这段广告内容是否违规，并说明理由",
  "input": "...",
  "output": "标签：违规\n理由：..."
}
```

SFT 让模型学习：

```text
用户问什么 → 应该按什么格式回答
```

### 1.3 RLHF：让模型符合“人类喜欢的回答”

有些回答事实正确，但用户不喜欢：

```text
太生硬
不解释原因
不承认不确定性
格式不可读
```

RLHF 通过偏好数据或奖励模型，优化模型输出风格和价值取向。

## 2. 内容理解任务里的 SFT：标签和理由不能一视同仁

腾讯广告内容理解实践指出，很多 NLU 任务可以抽象为：

```text
Prompt → 标签 + 理由
```

例如：

```text
输入：广告素材文本/图片描述
输出：是否违规 + 理由
```

普通 SFT 会把标签和理由当成同等 token 学习：

```text
标签 token 错一个，理由 token 错一个，损失上可能差不多
```

但业务上二者重要性不同：

```text
标签错了 = 业务决策错了
理由写得差 = 可解释性差
```

因此需要对标签更敏感。

### 2.1 RSCB：标签与理由解耦加权

可以理解为：

```text
标签部分损失权重大
理由部分损失权重相对小
```

伪代码：

```python
loss = label_weight * loss_on_label_tokens + reason_weight * loss_on_reason_tokens
```

这能让模型更关注最终决策。

### 2.2 PIF：负例 Feedback 也有价值

很多训练数据只告诉模型“正确答案是什么”，但没告诉它“为什么另一个选项错”。

PIF 的思想：

```text
把负例反馈也加入训练，让模型知道错误选项为什么错。
```

这对多项选择题尤其重要，因为模型容易有选择偏差（Selection Bias）：

```text
总偏好 A/B/C 某个位置
或者只看选项表面关键词
```

加入负例反馈后，模型能更好地区分相似选项。

## 3. RLHF 在 NLU 任务中为什么要改造？

通用 RLHF 常优化“回答看起来更好”：

```text
更礼貌
更流畅
更符合人类偏好
```

但 NLU 内容理解任务的核心目标是：

```text
标签必须正确
理由要能支持标签
```

如果奖励模型只看“理由写得好不好”，就可能出现：

```text
理由很漂亮，但标签错了
```

因此腾讯广告实践提出了更适合 NLU 的 RLLR（Label-Sensitive Reward）：

```text
奖励对标签更敏感，标签错时理由再好也不能高分。
```

直觉公式：

```text
Reward = 标签正确性奖励 + 理由质量奖励
但标签错误时，对总奖励施加强惩罚。
```

## 4. 垂域多模态模型：WeMM-Ecom 的训练经验

WeMM-Ecom-1.5 是微信电商场景的垂域多模态大模型。它提供了一个非常值得学习的范式：

> 不只是“拿大模型微调一下”，而是围绕业务目标重新设计数据、训练流程和评测体系。

### 4.1 数据：从规模驱动到质量与配方驱动

普通思路：数据越多越好。

WeMM-Ecom 的实践：

```text
数据多样性：扩充来源、图像聚类采样、丰富任务类型
数据准确性：用小规模消融实验和真实指标筛数据
```

构建约 2.6 亿条高质量训练语料。

### 4.2 训练流程：CPT + OPD + GRPO

```text
持续预训练 CPT：
  阶段1：知识注入预训练 —— 扩展知识广度
  阶段2：退火预训练 —— 聚焦高质量业务数据

后训练：
  用 OPD（在线策略蒸馏）替代传统 SFT
  再用 GRPO 做业务任务正确性对齐
```

为什么 OPD 能替代 SFT？

```text
传统 SFT：学固定答案
OPD：从更强策略/在线反馈中蒸馏行为，更像“动态老师”
```

### 4.3 评测体系：必须同时看通用能力和业务能力

WeMM-Ecom 评测集扩展到 65 个：

```text
38 个开源通用评测
27 个电商业务评测
```

这是垂域模型很重要的原则：

> 业务能力提升不能以通用能力崩塌为代价。

## 5. 从商户行业识别看多模态训练路线

商户行业识别场景中，输入可能包括：

```text
门头照
店内照
商户名称
结构化字段
```

这类任务适合多模态大模型，因为单一文本或单一图像都不够。

一条可复用路线：

```text
Step 1：用大模型直接推理，建立基线
Step 2：用较小模型做 SFT，降低推理成本
Step 3：用 GRPO 做强化精调，提高业务指标和泛化能力
```

这和 WeMM-Ecom 的经验一致：

```text
先用强模型探索上限
再把能力蒸馏/微调到小模型
最后用奖励函数对齐业务目标
```

## 6. 学完 Part4 后应形成的工程认知

```text
1. GPT 不是从“聊天”开始的，而是从预训练续写开始的。
2. SFT 解决“听指令”，但不能保证符合人类偏好或业务目标。
3. RLHF/GRPO 解决“偏好对齐”，但奖励函数必须和任务目标一致。
4. NLU 任务中，标签比理由更关键，需要标签敏感训练。
5. 垂域多模态模型的关键不是模型结构，而是数据配方、训练流程和评测体系。
6. 小模型如果训练得好，可以在垂域任务上超过更大的通用模型。
```
