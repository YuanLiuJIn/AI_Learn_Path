## 1. 文章解决了什么问题？

### 核心矛盾

```
现有 GUI Agent 的问题：
  ✅ 基础操作（点按钮、填表单）效果不错
  ❌ 遇到特定业务场景就歇菜：
     - 不认识公司 OA 的专属图标
     - 不知道 CRM 的审批流程是"流转"而非"提交"
     - 复杂表单的字段依赖关系不理解

SFT（监督微调）的瓶颈：
  - 需要海量标注数据（百万级）
  - 界面一变就得重新标注
  - 标注成本爆炸

本文核心答案：
  用 GRPO 强化学习 → 极少数据（3K）即可超越 SFT 效果
```

## 2. GRPO vs PPO vs SFT：三个关键对比

### 2.1 GRPO 为什么比 PPO 更适合 GUI Agent？

```
PPO：
  需要额外训练一个 Critic（价值网络）
  → 显存翻倍
  → "这一步到底值多少分？"很难定义（GUI 任务的信用分配问题）

GRPO：
  完全不要 Critic！
  → 给同一任务生成多个候选方案
  → 奖励高的优于组内平均 → 强化
  → 奖励低的低于组内平均 → 弱化
  → 显存省 50%，训练速度提升 30%+
```

### 2.2 GRPO 核心公式

```
Ai = (ri - mean({r1,...,rN})) / std({r1,...,rN})

ri：第 i 个候选响应的奖励
Ai > 0：这个方案比平均好 → 强化
Ai < 0：这个方案比平均差 → 弱化
```

### 2.3 GRPO 四大应用场景

| 场景 | 训练目标 | 代表论文 |
|---|---|---|
| **单步动作优化** | 点击/输入/滚动等原子操作精准 | UI-R1（136 样本即收敛） |
| **多步任务规划** | 长流程自主纠错 | GUI-R1（3K 样本） |
| **跨平台适配** | Windows/Android/Web 统一策略 | GUI-R1 统一动作空间 |
| **小样本学习** | 数据极度稀缺场景 | 仅需 136-3000 样本 |

## 3. 四大论文方案详解

### 3.1 GUI-R1：统一动作空间 + GRPO

**论文**：arxiv.org/pdf/2504.1...

**核心思路**：
- 把 Windows/Android/Web 的操作统一成一套规则
- 用统一奖励函数打分
- GRPO 优化策略

**奖励函数设计**：

```
最终奖励 = α × 格式奖励 + β × 准确性奖励

格式奖励：
  输出必须包含 <think>...</think> 和 <answer>...</answer>

准确性奖励：
  = 动作类型正确？(1/0)
  + 点击位置在目标区域内？(1/0)
  + 输入文本 F1 > 0.5？(1/0)
```

**数据构建策略**：
```
初始数据：1400 万条（5平台 × 3任务类型）
  ↓
筛选逻辑：用 Qwen2.5-VL-7B 生成 10 个候选，移除奖励为 0 或 1 的
  ↓
最终训练集：仅 3K 条（原数据的 0.02%）
  ↓
效果：超越 1300 万 SFT 数据的模型
```

### 3.2 Mobile-R1：三阶段渐进训练

**核心创新**：打破"机械执行单步操作" → 升级为"会规划、能探索、可纠错"

```
阶段 1：格式微调（SFT）
  → 学会标准输出格式
  → 理解"错误→修正"的逻辑
  → 4635 条轨迹数据

阶段 2：单步 GRPO
  → 优化每一步操作的准确性
  → 奖励：坐标命中 + 动作类型 + 格式

阶段 3：任务级 GRPO
  → 优化全局任务规划和纠错能力
  → 奖励：整个轨迹的连贯性 + 完成度
  → 允许试错：模型自己从错误中恢复

关键设计：
  阶段 3 的奖励只在任务结束后给！
  → 迫使模型从全局角度规划（而非贪心每步最优）
```

**Mobile-R1 的"灵光一闪"机制**：
```
正常路径：A → B → C → D（完成）
              ↘ E（误操作）
                → 返回 B → C → D（自纠正！）
                
传统训练：E 被当作"错误"惩罚
Mobile-R1：只要最终完成，中间纠错反而加分
  → 模型学会了"犯错后自我修复"
```

### 3.3 GUI-Critic-R1：术前批评机制

**核心创新**：在执行每个操作**之前**，先判断这个操作对不对。

```
类比：手术前的风险评估

普通 Agent：
  看到"删除"按钮 → 直接点 → 文件删了 → 晚矣

GUI-Critic-R1：
  看到"删除"按钮
  → Critic 先评估："这个操作正确吗？"
  → "当前任务是重命名，不是删除！"
  → 输出纠正建议："应该点击'重命名'而非'删除'"
  → Agent 调整操作 → 安全

训练方法：
  S-GRPO（Suggestion-aware GRPO）
  → 在标准 GRPO 基础上，额外加入"建议奖励"
  → 建议与正确答案相似 → 加分
```

### 3.4 MobileGUI-RL：在线自主探索学习

**核心创新**：不需要人工定义所有任务，Agent 自己探索生成训练任务。

```
Agent 自主探索：
  随机在 APP 中点击 → 记录轨迹 → GPT-4o 反推自然语言任务

例如：
  探索轨迹：打开相机 → 点击放大 → 点击拍照
  ↓ GPT-4o 反推
  任务描述："打开相机并放大后拍照"

任务筛选（文本世界模型）：
  LLM 模拟执行，能在最大步数内成功 → 保留为训练任务
  做不到 → 剔除
  
优势：
  ✅ 不需要人工写任务描述
  ✅ 自动发现长尾边缘场景
  ✅ 难度自适应递增
```

## 4. 企业落地四步实战

### Step 1：梳理 APP 核心场景与数据准备

```text
场景梳理：
  - 列出核心任务（如"电商下单""OA 审批"）
  - 列出关键界面元素（如"加入购物车按钮"）
  - 列出典型操作路径和常见错误

数据策略：
  - 优先收集"难样本"：基础模型做不对的操作
  - 参考数据量：136-3000 样本即可
  - 覆盖多样性：不同界面状态、弹窗、异常场景
```

### Step 2：设计 APP 专属奖励函数

```
准确性奖励：
  ✓ 动作类型匹配 → 1 分
  ✓ 坐标在目标区域内 → 1 分
  ✓ 文本 F1 > 0.8 → 1 分

效率奖励：
  ✓ 轨迹越短 → 奖励越高

安全奖励：
  ✓ 危险操作正确 → +0.3 分
  ✗ 危险操作错误 → -1 分

格式奖励：
  ✓ 输出含 <think> + <action> → +0.2 分
  ✗ 格式错误 → -0.5 分
```

### Step 3：分阶段训练

```
感知预训练(SFT) → 单步GRPO → 任务级GRPO
```

### Step 4：在线迭代

```
APP 更新 → 只补充差异界面样本 → GRPO 快速微调（无需全量重训）
线上错误 → 自动加入训练集 → 模型持续进化
```

## 6. 一句话总结

> GRPO 用"组内相对比较"代替了"训练独立 Critic"，用"规则奖励"代替了"人工标注"，用"极少量数据（136-3000 条）"替代了"SFT 的百万级样本"。这是目前 GUI Agent 特定场景训练的**性价比之王**。
